# CLAUDE.md — GlassBox

Guidance for Claude Code working in this repo.

## Project identity
GlassBox is a **spec-first** project. The **frontend** is generated by **Codeplain** from `***plain` specs — the source of truth is the `.plain` files, **not** the generated code. The **backend "brain"** (`server/`, Python FastAPI) is hand-written and proven; the frontend talks to it over HTTP.

## The golden rule
**Edit `glassbox.plain` (and `template/*.plain`), never the generated frontend code.** Codeplain overwrites the rendered output on every render, so hand-edits there are lost. If the rendered UI is wrong, fix the corresponding `.plain` spec and re-render.

## What renders, what doesn't
- `glassbox.plain` → the **React frontend** (the dashboard: enter a goal → watch the Bull/Bear debate → verdict-hero + Signal Strength → "Prove it" → "Verify" MATCH → edit the signed record → TAMPER DETECTED (interactive tamper)). `resources/ui_reference.md` is the design brief the renderer should follow; `server/glassbox/static/index.html` is the working reference implementation + demo fallback.
- `server/` (Python FastAPI) is **hand-written, not rendered** — the multi-agent brain, the relevance gate (`guard.py`), live CoinGecko + DeepBook feeds, ed25519 signing, Walrus write (+ on-chain Sui object), verify, the standalone `verify_cli`, and the `DEMO_MODE` cache. The frontend calls it at `VITE_API_BASE_URL`.
- Generated output dirs (`web/`, `plain_modules/`, `dist/`, `conformance_tests/`, `.tmp/`) are **gitignored build artifacts**.

## Render workflow
```bash
# author: edit glassbox.plain / template/*.plain
codeplain glassbox.plain --dry-run                       # validate the spec
CODEPLAIN_API_KEY=<key> codeplain glassbox.plain --force-render --headless
# (then run any post-render patch script if present, and `npm run dev`)
```
Use promo code **London100** for 100 free render credits. If a single functional spec is too large for the renderer, split it into smaller items (FRIDs).

## Backend contract (keep both sides in sync)
The frontend and brain agree on the HTTP shape: `POST /api/analyze {goalText, asset, risk}` → Decision (off-topic input → `422 {outOfScope, message}`); `POST /api/audit {decision, goalText}` → AuditRecord (recordHash, signature, pubkey, blobId, **suiObjectId, anchorEpoch, recordCanonical**); `GET /api/verify/{recordId}` → {hashMatch, signatureValid}; `POST /api/rehash {decision}` → {recordHash}; `GET /api/health`; `GET /api/pubkey`. The UI's interactive tamper demo hashes `recordCanonical` client-side (Web Crypto). When you change one side, update the other.

## Demo mode
`DEMO_MODE=1` makes the canonical pitch question return a pre-baked Decision instantly (bake it with `python3 -m glassbox.seed_demo`); other questions fall through to live. See `DEMO.md` for the stage run-sheet. Tests: `cd server && python3 -m pytest -q` (100, mocked; also CI); `python3 -m glassbox.usecases` runs a live gallery of real queries. Use a venv (`requirements.txt` + `requirements-dev.txt`).

## Claim discipline (everywhere — UI, README, deck)
Say **"tamper-evident"**, never "tamper-proof". The **signature proves origin**; the **Walrus-registered on-chain Sui object** is the anchor — an independent on-chain reference (a Sui *epoch*, not a precise wall-clock time — don't overclaim). It is an **evidence layer**, not model validation or a compliance guarantee. Never pitch returns/PnL.
